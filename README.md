This project is a mini expression compiler written completely in Java. The purpose of the project is to simulate the basic phases of how a real compiler works. Instead of compiling an entire programming language, my program focuses on arithmetic expressions. The compiler takes a mathematical expression entered by the user, breaks it into tokens, checks if the expression follows proper grammar rules, builds a syntax tree, and then evaluates the final result.
The first part of the project is the lexical analyzer, also called the tokenizer. The lexer reads the input expression character by character and converts it into meaningful tokens. Tokens are categories such as numbers, operators, and parentheses. For example, if the user enters:
  (3 + 2) * 5 - 1
the lexer converts it into:
  ( 3 + 2 ) * 5 - 1
I used an enum called TokenType to define all possible token categories like NUMBER, PLUS, MINUS, MULTIPLY, DIVIDE, LPAREN, and RPAREN. I created a Token class to store both the token type and its actual value. This makes the parser easier to work with later because each part of the expression already has a label attached to it.
Inside the lexer, I used a while loop to scan through the input string until the end is reached. If the current character is a digit, the program continues collecting digits so it can handle multi-digit numbers like 123 instead of treating them as separate numbers. If the character is an operator or parenthesis, the lexer creates the appropriate token and moves to the next character. If an invalid character is found, the program throws an error message.
The next phase is parsing. The parser checks whether the token stream follows the correct grammar rules for arithmetic expressions. I used recursive descent parsing because it is one of the simplest ways to implement a parser manually in Java. The grammar I followed was:
  E → T ((+ | -) T)*T → F ((* | /) F)*F → NUMBER | (E) | -F
The reason I separated the grammar into expressions, terms, and factors is to automatically handle operator precedence. Multiplication and division should happen before addition and subtraction, and this grammar structure makes that happen naturally.
I implemented separate methods called parseExpression(), parseTerm(), and parseFactor(). Each method handles a different part of the grammar. The parser works recursively, especially when parentheses are involved. For example, when the parser sees a left parenthesis, it recursively parses the entire expression inside the parentheses before continuing.
While parsing, the program also builds an Abstract Syntax Tree, or AST. The AST is a tree representation of the expression. Each node in the tree represents either a number or an operator. I created an abstract class called ASTNode and then created subclasses such as NumberNode, BinaryOperationNode, and UnaryMinusNode.

This tree structure is important because it preserves the correct order of operations. The evaluator then recursively traverses the tree from the bottom up to calculate the final answer.
For evaluation, each AST node has an evaluate() method. A NumberNode simply returns its value, while a BinaryOperationNode evaluates its left and right children and then applies the correct operation. I also included support for unary minus so expressions like -3 + 5 work correctly.
I added exception handling throughout the program to detect invalid expressions. For example, inputs such as:
  3 + * 5    or    3 + (4 - )
will produce clear error messages instead of crashing the program.
One of the main ideas behind my design was separating the project into phases just like a real compiler. The lexer only focuses on tokenization, the parser only focuses on grammar validation and tree construction, and the evaluator only focuses on computing the result. Using object-oriented programming made the project more organized and easier to expand in the future.
Overall, this project helped me better understand compiler design concepts such as lexical analysis, recursive descent parsing, context-free grammars, recursion, abstract syntax trees, and expression evaluation. Even though this compiler only works with arithmetic expressions, it demonstrates the same core ideas that larger programming language compilers use.
